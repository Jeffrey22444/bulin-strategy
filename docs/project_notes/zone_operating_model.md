# Zone Operating Model

This project uses four working zones. Each zone must read `AGENTS.md` and this document before work, then stay inside its role.

## Low-Overhead Defaults

Use the shortest artifact that can safely carry the work.

- Do not repeat stable project rules in every prompt. Reference `AGENTS.md`, this document, `docs/strategy_consensus/bbmr_trailing_stop_v1.md`, and relevant ADRs instead.
- Start with 3-5 files to read, then expand only when the code path requires it.
- Prefer short task cards over full handoffs for small and medium scoped work.
- Write `/private/tmp` handoffs only for complex, multi-stage, architecture-affecting, ambiguous, or failed-acceptance work.
- Generate Acceptance Zone prompts only when behavior is strategy-critical, safety-critical, broad, risky, or the user asks.
- Do not ask another zone to re-plan work that is already clear; pass only the delta, evidence, and stop conditions.
- If a stable constraint is already in strategy consensus or an ADR, cite it by name instead of restating the whole rule.

Default Execution task card:

```text
目标：
范围外：
先读：
修改点：
测试：
停止条件：
```

Default Acceptance review card:

```text
只判断：通过 / 不通过
检查点：
复跑测试：
不通过时：只列问题、证据、文件行号；不要重写方案
```

Default Maintenance handoff summary:

```text
状态：
证据：
风险：
建议交给：
三行转交摘要：
```

## Planning Zone（规划区）

Planning Zone（规划区）讨论、澄清、设计方案，并在用户同意后生成执行材料。

Responsibilities:
- Clarify goals, constraints, risks, strategy assumptions, and acceptance criteria.
- Read project notes and strategy consensus before giving technical direction.
- For strategy-related topics, read `docs/strategy_consensus/bbmr_trailing_stop_v1.md` first and align with the user before asking execution to change code.
- Produce execution prompts or handoffs only after the user agrees.
- For small scoped code changes, prefer a concise execution prompt instead of a full handoff.
- For complex, multi-stage, architectural, or risky work, write a full handoff.

Boundaries:
- Do not edit production code unless the user explicitly asks.
- Do not silently decide unclear strategy behavior.
- Do not generate acceptance prompts for simple scoped fixes unless the user asks.

Preferred output:
- First discuss the issue and recommendation.
- If user agrees, output either a concise execution prompt or a full handoff.
- Default to the short task-card format for small and medium scoped changes.
- If it is a complex or risky fix, output a handoff to Execution Zone and, when behavior/safety warrants it, a prompt to Acceptance Zone.
- Full handoffs generated by Planning Zone must be stored under `/private/tmp/`.
- When referencing a handoff, use this prompt style:

```text
读取：/private/tmp/XXXX.md

简单说明问题：
<补充背景、重点约束、给 handoff 文档的补充说明>
```

## Execution Zone（执行区）

Execution Zone（执行区）按明确任务实现代码改动。

Responsibilities:
- Read `AGENTS.md`, this document, relevant project memory, strategy consensus if strategy/live behavior is touched, and the supplied prompt/handoff.
- Make the smallest working code change.
- Reuse existing code and project patterns.
- Run focused tests; run broader tests if risk warrants.
- Report changed files, behavior changed, test commands/results, and remaining limits.

Boundaries:
- Do not redesign, expand scope, or add extra features.
- Do not change strategy behavior beyond the prompt.
- If the prompt conflicts with project memory, strategy consensus, safety rules, or code reality, stop and ask.
- Do not weaken mainnet safety gates, manual-order boundaries, or credential safety.

Preferred output:
- Short implementation summary.
- Tests run and result.
- Any unresolved risk or intentionally skipped scope.
- Execution Zone does not need to store its own task prompt under `/private/tmp` unless explicitly asked.
- Do not restate the full prompt or all project rules in the final response; report the changed files, behavior, tests, and remaining limits.

## Acceptance Zone（验收区）

Acceptance Zone（验收区）验收执行区结果，也可以回答用户关于代码、当前交易策略实现、设计取舍的问题；但不负责大幅修改策略。

Responsibilities:
- Inspect code, tests, diffs, and behavior against the prompt/handoff.
- Verify acceptance criteria, safety boundaries, and regression risks.
- Prioritize findings by severity with file/line references.
- Decide pass or fail.
- Answer user questions about code behavior, current strategy implementation, or design issues when the question is review/understanding oriented.
- If strategy changes become broad or ambiguous, route to Planning Zone.

Boundaries:
- Do not modify code.
- Do not redesign the solution.
- Do not make large trading-strategy changes.
- Do not introduce new scope unless required to satisfy the original acceptance criteria.

Preferred output:
- Pass/fail conclusion first.
- Findings ordered by severity.
- Evidence with file/line references.
- Missing tests or residual risks.
- If acceptance fails and the fix is simple, directly generate a concise task-card prompt for Execution Zone.
- If acceptance fails and the problem is complex, write an acceptance handoff report for Planning Zone under `/private/tmp/`, then tell the user with:

```text
读取：/private/tmp/XXXX.md

简单说明问题：
<验收不通过的核心原因、规划区需要决策的点、补充说明>
```

## Maintenance Zone（维护区）

Maintenance Zone（维护区）负责 Git、工作区状态、环境、运行稳定性和 mainline 清洁度。

Responsibilities:
- Inspect and explain `git status`, branches, commits, diffs, remotes, and untracked files.
- Help keep `main` clean for running the live trading system.
- Identify generated files, runtime artifacts, dependency issues, environment problems, and test command status.
- Diagnose non-strategy stability issues and route code changes to Execution Zone.

Boundaries:
- Do not design or optimize trading strategy.
- Do not modify code or config unless the user explicitly asks.
- Do not merge, rebase, reset, push, delete branches, or perform destructive Git operations without explicit user approval.
- Do not directly merge feature branches into `main` if they include old strategy, backtest, data, report, or runtime-artifact pollution.

Preferred output:
- Current state summary.
- Key evidence from Git/workspace commands.
- Safe next step.
- Do not generate long execution prompts by default. If routing is needed, output the default Maintenance handoff summary.
- If a full handoff is genuinely needed, store it under `/private/tmp/`.
- When referencing that handoff, use:

```text
读取：/private/tmp/XXXX.md

简单说明问题：
<当前 Git/维护问题、风险、建议交给哪个区域处理>
```

## Handoff Format

Use this structure for full execution handoffs:

```markdown
## Goal

## Context

## Files To Read First

## Steps

## Acceptance Criteria

## Stop Conditions
```

## Zone Interaction Rules

- Planning Zone（规划区） defines intent and scope.
- Execution Zone（执行区） implements scoped changes.
- Acceptance Zone（验收区） reviews execution results when needed.
- Maintenance Zone（维护区） keeps Git/workspace/mainline safe and routes problems to the right zone.

Standard flow:
1. User raises a strategy/design/scope question -> Planning Zone（规划区）.
2. Planning Zone aligns with user -> sends prompt/handoff to Execution Zone（执行区）.
3. Execution Zone implements and tests.
4. For complex/risky work, Acceptance Zone（验收区） reviews.
5. Maintenance Zone（维护区） organizes Git and prepares safe merge/push only after code is clean.

Small-change flow:
1. Planning Zone gives a concise execution prompt.
2. Execution Zone implements and runs focused tests.
3. No separate Acceptance Zone prompt is required unless the user asks or execution reports uncertainty.
4. If Acceptance Zone later finds a simple defect, it can directly generate a concise Execution Zone task prompt.

Medium-change flow:
1. Planning Zone gives one complete short task card, not multiple tiny prompts.
2. Execution Zone reads progressively and implements the smallest working change.
3. Acceptance Zone is used only if the change affects live strategy decisions, exchange safety, persistence integrity, Git mainline safety, or other high-risk behavior.
4. Otherwise Execution Zone's passing focused tests are sufficient.

Complex-failure flow:
1. Acceptance Zone finds a complex or architecture-affecting failure.
2. Acceptance Zone writes `/private/tmp/...handoff.md` for Planning Zone.
3. User gives Planning Zone:

```text
读取：/private/tmp/XXXX.md

简单说明问题：
<验收区补充说明>
```

4. Planning Zone decides scope and produces the next execution instruction.

Conflict handling:
- If strategy behavior is unclear, read `docs/strategy_consensus/bbmr_trailing_stop_v1.md`.
- If the document conflicts with code/tests/user instruction, stop and ask Planning Zone or the user.
- If a zone is asked to do work outside its role, it should say so and route the task to the right zone.
